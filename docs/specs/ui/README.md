# UI仕様

## 共通

- レスポンシブデザイン
- デスクトップ、ノート PC、MacBook、タブレット、スマホの詳細なレスポンシブ仕様は `../../designs/responsive/README.md` に従う
- スマホ固有の詳細仕様は `../../designs/responsive/mobile.md` に従い、スマホではハンバーガーメニュー、ボトムナビ、1カラム表示、全画面モーダルを基本にする
- 明確なステータス表示
- 情報はカード・テーブル形式で整理
- 一覧画面に詳細パネルを同時表示せず、記事選択時は詳細画面へ遷移
- 主要画面は Vue Router の browser history route として `/login`、`/register`、`/articles`、`/articles/:id`、`/calendar`、`/tags`、`/settings` を持つ
- 認証済みで `/login` / `/register` を開いた場合は router guard で `/articles` へ戻し、未認証で protected route を開いた場合は `/login` へ戻す
- UI 表示言語は日本語 / English に対応する
- 初期表示言語は `localStorage` の `articleshelf.locale`、ブラウザ言語、英語フォールバックの順で決定する
- 未対応ブラウザ言語、未対応 locale、未定義翻訳キーは英語へフォールバックする
- 左サイドバー下部ではユーザー名の下に言語切替トグルを表示し、日本語 / English を切り替えられるようにする。選択結果は `localStorage` に保存する
- `960px` 未満では左サイドバーを常時表示せず、上部ハンバーガーから一時ドロワーを開く。ドロワーには一覧フィルタ、カレンダー、タグ管理、アカウント、ログアウト、言語切替を表示する
- 記事詳細画面でも同じ workspace shell を維持し、desktop では sidebar、mobile では上部ハンバーガー経由の drawer を使って一覧 / カレンダー / タグ管理 / アカウント / ログアウトへ移動できる
- `599px` 以下では下部に未読 / すべて / お気に入り / カレンダー / 追加のボトムナビを表示する。詳細画面、追加・フィルタ・確認・アカウント系モーダル表示中、未保存確認中はボトムナビを表示しない
- UIフォントは `Noto Sans JP`、`M PLUS 1p`、日本語OS標準ゴシック、system-ui の順にフォールバックする
- コード表示は `JetBrains Mono`、`Fira Code`、`Cascadia Code`、OS標準等幅フォントの順にフォールバックする
- Vuetify のUIロケールは現在の表示言語に追従し、カレンダーや入力補助UIも日本語 / 英語で表示する
- 画面上の日付表記は現在の表示言語に応じて `ja-JP` / `en-US` で表示する。APIや永続化で扱う日付値は既存契約に合わせて `YYYY-MM-DD` を維持する
- 日付ピッカーの上部は年・月の順で表示し、曜日と日付は日曜を赤系、土曜を青系で表示する
- 新規作成・編集・絞り込みなど操作完了を伴うモーダルは、タイトルと主要アクションを同じヘッダー行に配置し、本文末尾に同じアクションを重複配置しない
- モーダルのヘッダー行は左にタイトル、右に閉じる・保存・適用などのアクションを置き、モバイルでは縦積みでボタンを押しやすくする
- 追加、フィルタ、アカウント設定など入力量の多いモーダルは `599px` 以下で全画面表示にし、確認ダイアログやカレンダーの日別記事一覧は下端寄せの bottom sheet として表示する

## 記事一覧画面

- 検索バー、並び替え、フィルタ、記事追加の導線を表示する
- 未読 / 既読 / すべて / お気に入りの絞り込みを提供する
- タグ、おすすめ度、登録日範囲、既読日範囲で絞り込める
- 複数タグを選んだ場合は OR 条件で、いずれか1つ以上のタグを持つ記事を表示する
- 適用中のフィルタ条件は一覧画面上で確認できるようにする
- 一覧画面のタイトルは左サイドバーや上部フィルタで選択中の条件に合わせて切り替える
- 一覧画面では current page の記事だけを表示し、前のページ / 次のページ導線で server-driven query の page を移動できるようにする
- 記事カードから詳細表示、既読切り替え、お気に入り切り替え、削除ができる
- 未読から既読へ切り替えると既読日は操作日になり、既読から未読へ戻すと既読日は未設定になる
- ステータス切り替え後は、スナックバーで結果と元に戻す導線を表示する
- 一覧取得、記事詳細取得、一覧削除に失敗した場合は、一覧画面内で再試行可能なエラー表示を行う
- 記事数が多い場合でも主要操作を継続できる
- `GET /api/articles` の一覧 query では `status`, `search`, `favorite`, repeatable `tag`, repeatable `rating`, `createdFrom`, `createdTo`, `readFrom`, `readTo`, `sort`, `page`, `size` を使い、記事一覧の検索 / 絞り込み / 並び順 / ページ取得を server-driven に反映する
- カレンダー、タグ管理からの記事一覧導線や左サイドバー件数は、当面の互換として別取得の全件 snapshot を使う
- 左サイドバー下部に学習継続を促すメッセージを表示し、画面遷移や記事追加などのタイミングでローテーションする

記事カードのタイトル省略、タグ表示件数、OGP画像、日付・ステータス配置、スクロールバー、ヘッダー操作群の整列などの見た目は [Article Components](../../designs/components/articles.md) を正本とする。

## カレンダー画面

- 左サイドバーにカレンダー画面への導線を表示する
- カレンダー画面の上部に表示月、前月 / 次月ボタンを表示する
- カレンダーのタイトルと登録日 / 既読日のトグルは同じ行に配置する
- 登録日 / 既読日の表示モードをトグルで切り替え、トグルは詳細画面の閲覧 / 編集トグルと同じ共通スタイルを使う
- 表示月の前月 / 次月操作は登録日 / 既読日のトグルと同じ操作行に配置し、行内中央に表示する
- カレンダー操作行の主要要素は高さを揃える
- 月内の追加数、既読数、積読差分をカレンダー上部に表示する
- 月内の追加数、既読数、積読差分はサマリー全体の囲い内で中央寄せにし、個別項目には囲いを付けず、最も長い表示に合わせた同じ最小幅で表示し、数値はラベルの右側に表示する
- 曜日見出しと日付セルは土曜、日曜で色分けする
- 日付セルは6週分の固定グリッドとして表示し、月移動や記事数の違いで高さが変わらず、カレンダー領域の下端まで揃うようにする
- 日付セルには該当日の記事数と記事タイトルを表示する
- 日付セル内の記事タイトルはクリック可能にし、記事詳細へ遷移する
- 記事数が多い日付セルはセル内でスクロールする
- カレンダー本体と日付セル内のスクロールバーは控えめな薄い表示にする
- スマホ幅では日付セル内の記事表示を最大1件程度に抑え、日付セルタップでその日の記事一覧 bottom sheet を開く。bottom sheet 内の記事を選ぶと記事詳細へ遷移する

## 記事詳細画面

- 記事ヘッダーにタイトル、URL、登録日時、更新日時を表示する
- 通常は閲覧モードで表示し、閲覧 / 編集トグルで編集モードへ切り替える
- 閲覧モードでは概要、タグ、メモ、ステータス、既読日、おすすめ度を確認できる
- 閲覧モードでメモが未入力の場合、メモ欄には「メモはまだありません」と表示する
- 閲覧モードのメモは Markdown として表示し、コードブロックのシンタックスハイライト、ファイル名表示、行番号表示に対応する
- Markdown 表示では raw HTML を無効化し、サニタイズ済み HTML だけを描画する。リンクは `http` / `https` / `mailto`、画像は `http` / `https` のみ許可し、コードブロックは実行せず静的にハイライトする
- 編集モードではタイトル、URL、概要、タグ、メモ、ステータス、既読日、おすすめ度を編集できる
- 更新保存では `GET /api/articles/{id}` から受け取った `version` を `PUT /api/articles/{id}` へ送り、他タブ・他端末更新と競合した場合は `ARTICLE_VERSION_CONFLICT` を受け取る
- 編集モードでステータスを未読から既読へ変更した場合、既読日が未設定なら操作日を入れる。既存の既読日がある場合は維持する
- 編集モードでステータスを既読から未読へ変更した場合、既読日は未設定にする
- メモ編集では Markdown の表示結果を保存前に確認できる
- 編集途中の内容は閲覧 / 編集トグルを切り替えても保持し、閲覧モードのメモ表示にも保存前のドラフトを反映する
- 閲覧モード中もお気に入りはヘッダーから切り替えできる
- 保存中は二重送信を防止する
- 未保存の差分がある状態で別画面へ移動しようとした場合は、画面内操作、ブラウザ履歴、URL 変更のいずれでも、破棄して移動するか編集を続けるかを確認する警告ダイアログを表示する
- 保存や削除に失敗した場合は詳細画面内にエラーメッセージを表示し、編集内容を保持したまま再試行できるようにする
- `ARTICLE_VERSION_CONFLICT` の場合は、保存中のドラフトを残したまま競合メッセージと「最新を読み直す」導線を表示し、利用者が最新版を再取得してから差分を反映し直せるようにする
- 削除操作では確認ダイアログを表示し、確定後に削除する
- 一覧起点で開いた詳細では、左サイドバーや mobile drawer の active 状態は現在の一覧フィルタ文脈を保つ。カレンダー起点で開いた詳細ではカレンダーを active に保ち、戻るボタンで同じ月・表示モードへ戻る

詳細ヘッダー、閲覧 / 編集トグル、右側メタ情報、タグ編集、メモ入力、Markdown 表示領域、アイコンボタン、幅やスクロールの見た目は [Article Components](../../designs/components/articles.md) を正本とする。

## 追加モーダル

- 初期表示では URL 入力、プレビュー、保存、閉じるを主役にし、タイトル、タグ、メモ、既読日、おすすめ度、あとで読むは「タグ・メモ・おすすめ度を追加」から展開する
- モーダルを開いた直後は URL 入力欄に初期フォーカスする
- URL は必須項目として扱い、未入力時は保存せずエラーメッセージを表示する
- URL の貼り付け時と入力欄 blur 時に、`http` / `https` の構文として妥当な URL だけ `POST /api/articles/preview` でプレビューを取得する。構文不正や未対応 scheme の URL では preview API を呼ばず、URL 入力のエラーを表示する。自動 debounce 入力監視は行わない
- URL が変更された場合、既存の preview / duplicate state は古いものとして扱い、保存可否や表示判断に使わない
- preview API でプレビューを取得できない場合は「プレビューを取得できませんでした。URLによっては保存時にエラーになる場合があります。」と表示し、保存時に正式な URL 確認を行う
- タイトルは任意項目として扱い、未入力時は URL から自動設定する
- 「あとで読む」を外して既読として保存する場合、既読日未入力では保存せずエラーメッセージを表示する
- 選択済みタグの削除、過去のタグからの追加、新しいタグの直接入力に対応する
- 既存タグからタグを追加した後は、既存タグ選択欄の値と検索文字列をクリアする
- ステータス選択は表示せず、「あとで読む」チェックボックスで未読保存を指定する
- 「あとで読む」はデフォルトでチェック済みとする
- おすすめ度は 0 から 5 の星で登録する
- 保存中は重複操作を抑止する
- 閉じるは URL バリデーションエラー表示後でもモーダルを閉じる
- 保存に失敗した場合は追加モーダル内にエラーメッセージを表示し、入力内容を保持する
- preview または保存で重複 URL が分かった場合は、登録済みの記事詳細へ移動する導線を表示する。preview duplicate 時は保存ボタンを非活性にし、URL 変更時に duplicate state を解除する
- 保存後は一覧画面に留まり、自動では詳細画面へ遷移しない
- 保存成功後は snackbar で保存完了を表示し、主 action「詳細を編集」は作成済み記事の詳細画面へ遷移、副 action「続けて保存」は新規状態で追加モーダルを再表示する

追加モーダルの幅、ヘッダー配置、入力ブロックの区切り、タグ入力、おすすめ度入力、日付入力、保存 / 閉じるボタン配置などの見た目は [Article Components](../../designs/components/articles.md) を正本とする。

## 認証画面

- デスクトップ幅では、左側に ArticleShelf の価値訴求、保存・タグ・進捗の説明、機能イメージカードを表示し、右側にログイン / ユーザー登録フォームを表示する
- 左側の機能イメージカードは、最近保存した記事を中心に、進捗カードを上、人気タグカードを下へずらして配置し、各カードの見出しと内容が読める程度の重なりに抑える
- タブレット幅以下では、機能イメージカードを省略し、ブランドコピーとフォームを縦に配置する
- フォームカードは背景より少し明るい面、薄い border、やわらかい shadow、丸みのある角で自然に分離し、主要操作領域として視認できる状態を保つ
- ログイン / 登録切り替えでは、補助説明文を重ねず、ラベルとボタン表示だけで操作の意味が伝わる簡潔さを保つ
- ログイン / ユーザー登録では、ユーザー名、表示名、パスワードの入力エラーを送信前に画面上で確認できるようにする
- ユーザー名は 3〜32文字の英小文字・数字・ドット・アンダースコア・ハイフンとして扱い、入力時は小文字へ正規化する
- ユーザー登録時の表示名は任意とし、未入力の場合は正規化済みユーザー名を表示名として使う。フォーム上では「任意」と表示し、未入力時の扱いをヒントで示す
- 入力欄の項目名は入力欄外の固定ラベルとして表示し、ブラウザの自動入力時に入力値と重ならないようにする
- 入力欄のプレースホルダーは入力例に限定し、ユーザー登録時のユーザー名やパスワードの入力ルールは欄下のヒントとして表示する
- 認証処理中は送信ボタンをローディング表示にし、二重送信を防止する
- ログイン後のサイドバーにはアカウント設定導線を表示する
- アカウント設定では、パスワード変更、全端末ログアウト、退会を扱う
- パスワード変更、全端末ログアウト、退会に成功した場合はログイン画面へ戻る

## タグ管理画面

- タグ管理画面では、登録タグ数、タグ名検索、並び替え、タグ名、紐づく記事数、編集、統合、削除を表示する
- タグ管理画面の上部には、記事一覧画面と同様に検索、並び替え、追加の操作をタイトルと同じヘッダー行に横並びで配置する
- 並び替えは `並び順` ラベル付きのセレクトとして表示し、選択値が日本語 / English のどちらでも見切れない幅を確保する
- タグ追加は「タグを追加」ボタンからダイアログを開き、タグ名を入力して追加する
- タグ追加ダイアログでは、タグ名入力欄の右端と追加ボタンの右端を揃える
- 並び替えは、記事数が多い順、記事数が少ない順、名前順を選択できる
- 編集、統合、削除の操作にはツールチップを表示し、初見でも操作内容が分かるようにする
- 使用中のタグは削除できない。削除ボタンは非活性にし、ツールチップで理由を表示する
- 未使用タグ削除の確認文では、タグだけが削除され、記事自体は削除されないことを表示する
- タグ名または記事数から、そのタグで絞り込んだ記事一覧へ移動できる
- タグが0件の場合は空状態を表示し、最初のタグを追加する導線を表示する

## 一覧取得エラー

- 記事一覧の初回取得や再取得に失敗した場合は、一覧上部にエラーバナーを表示する
- エラーバナーには再試行ボタンを表示し、その場で再読込できるようにする
- 記事カードからの記事詳細取得や一覧削除に失敗した場合も、詳細画面へ遷移せず一覧上部のエラーバナーへ表示する
- API が `messages` を返した 4xx はその文言を表示し、重複 URL の場合は登録済み記事へ移動する導線を維持する
- 401 / 403 / 404 で JSON エラー本文がない場合は、現在の表示言語に応じた汎用メッセージを表示する
- 通信失敗は接続確認メッセージ、5xx と malformed success response は内部詳細を出さない汎用メッセージとして表示する
