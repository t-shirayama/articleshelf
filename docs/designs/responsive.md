# レスポンシブ対応デザイン仕様

最終更新: 2026-05-09

## 1. 目的

ArticleShelf をデスクトップ、ノート PC、MacBook、タブレット、スマホで破綻なく使えるようにする。
1920x1080 のデスクトップ表示を基準にしつつ、画面幅と高さが小さい環境では余白、操作群、カード、詳細、モーダルを段階的に再配置する。

## 2. 対象サイズ

| サイズ | 想定環境 | 主な確認観点 |
| --- | --- | --- |
| `1920x1080` | デスクトップ代表 | 既存の余白、左サイドバー、一覧密度を維持する |
| `1440x900` | MacBook 代表 | サイドバーと一覧、詳細の高さが窮屈にならない |
| `1366x768` | ノート PC 代表 | 検索・フィルタ・追加ボタン、カード、学習継続カードが見切れない |
| `820x1180` | タブレット代表 | 一時ドロワー、1カラム一覧、広めモーダルが自然に使える |
| `430x932` | 大きめスマホ代表 | ボトムナビ、全画面モーダル、詳細縦積みが収まる |
| `390x844` | 標準スマホ代表 | 主要操作が重ならず、一覧カードが読みやすい |
| `375x667` | 小さめスマホ代表 | ヘッダー、下部ナビ、詳細操作、入力欄が見切れない |

## 3. ブレークポイント方針

| 幅 | レイアウト |
| --- | --- |
| `1181px以上` | デスクトップ。左サイドバーを常時表示し、記事一覧は固定高さのスクロール領域にする |
| `1180px以下` | コンパクトデスクトップ。余白とサイドバー幅を抑え、検索・操作群は必要に応じて折り返す |
| `960px未満` | タブレット / モバイル共通。左サイドバーを隠し、ハンバーガーから一時ドロワーを開く |
| `600px未満` | スマホ。下部ボトムナビを表示し、モーダルは全画面または下寄せシートにする |
| `380px以下` | 小型スマホ。ヘッダー、下部ナビ、詳細操作、ボタン間隔をさらに圧縮する |

高さが `900px` 以下のデスクトップでは、画面幅が十分でも縦方向の圧迫を避けるため、シェル余白、サイドバー内の間隔、記事カード高さ、詳細メモ欄の高さを抑える。

## 4. 画面別ルール

### 4.1 アプリシェル / ナビゲーション

- デスクトップでは左サイドバーを常時表示し、一覧領域と上下 padding を揃える。
- コンパクトデスクトップではサイドバー幅、シェル余白、学習継続カードの余白を縮める。
- `960px` 未満では左サイドバーを非表示にし、上部ハンバーガーと一時ドロワーに切り替える。
- `600px` 未満ではボトムナビを固定表示し、一覧末尾やボタンが隠れないよう下 padding と safe area を確保する。
- 小型スマホではブランドコピー、下部ナビラベル、アイコンボタンの間隔を詰めても、タップ領域は読み取れる状態を保つ。

### 4.2 記事一覧

- デスクトップではサムネイル左、本文右の2カラムカードを維持する。
- コンパクトデスクトップでは検索・並び替え・フィルタ・追加を折り返し可能にし、各ボタンの右端や文字が切れないようにする。
- タブレット / スマホではカードを1カラム化し、OGP サムネイルは横長比率を維持しつつ過大にならない高さに抑える。
- スマホでは検索入力、並び替え、フィルタ、追加ボタンを縦積みにし、主要アクションを見失わない順序にする。
- カード内のタイトル、概要、タグ、日付、ステータス、操作ボタンは重ならないよう、狭い幅ではメタ情報を折り返す。

### 4.3 記事詳細

- デスクトップでは本文と右メタカラムの2カラムを維持する。
- コンパクトデスクトップでは詳細幅とメタカラム幅を抑え、メモ編集欄の高さを画面高に合わせて調整する。
- `960px` 未満では詳細本文とメタ情報を1カラム化し、メタブロックはタブレットでは横並び、スマホでは縦積みにする。
- スマホでは戻る、閲覧 / 編集、保存、削除、お気に入りが重ならないよう、上部操作を複数行に分ける。
- 詳細画面が長くなる場合は、パネル内スクロールではなくブラウザ全体でスクロールする。

### 4.4 モーダル / ダイアログ

- デスクトップでは読みやすい固定幅のダイアログとして表示する。
- タブレットでは画面幅いっぱいに近い広めのダイアログを許容する。
- スマホでは記事追加、フィルタ、アカウント設定を全画面モーダルにする。
- 削除確認、タグ操作、カレンダー日別記事一覧はスマホでは下寄せシートとして表示し、下部 safe area を確保する。
- タイトル行と主要アクションは同じヘッダーに置き、本文末尾に同じ保存 / 適用操作を重複配置しない。

### 4.5 カレンダー / タグ管理

- カレンダーはデスクトップで6週固定グリッドを維持し、タブレット / スマホでも月移動と表示モード切り替えが重ならないようにする。
- スマホのカレンダーセルは記事タイトルを詰め込みすぎず、タップで日別記事一覧シートを開く。
- タグ管理はデスクトップでヘッダー操作群と一覧の右端を揃える。
- タブレット / スマホでは検索、並び替え、タグ追加を縦積みにし、行内編集や統合 / 削除操作が横にあふれないようにする。

## 5. スクリーンショット確認

公式デザインスクリーンショットは `docs/designs/screenshots/capture-designs/<viewport>/` に保存する。README の代表画像には `1920x1080` の画像だけを使う。
レスポンシブ崩れ確認用のサイズ別スクリーンショットは、公式画像とは分けて `frontend/test-results/responsive-screenshots/<command>/<viewport>/` に出力する。

実行コマンド:

- `npm run capture:designs`: `docs/designs/screenshots/capture-designs/1920x1080/` に公式デザインスクリーンショットを更新する
- `npm run capture:responsive`: `frontend/test-results/responsive-screenshots/capture-responsive-all/` に標準確認サイズをまとめて撮影する
- `npm run capture:responsive:laptop`: `frontend/test-results/responsive-screenshots/capture-responsive-laptop/` に `1366x768` を撮影する
- `npm run capture:responsive:macbook`: `frontend/test-results/responsive-screenshots/capture-responsive-macbook/` に `1440x900` を撮影する
- `npm run capture:responsive:tablet`: `frontend/test-results/responsive-screenshots/capture-responsive-tablet/` に `820x1180` を撮影する
- `npm run capture:responsive:mobile`: `frontend/test-results/responsive-screenshots/capture-responsive-mobile/` に `430x932`、`390x844`、`375x667` を撮影する

対象画面だけ確認する場合は、`ARTICLESHELF_SCREENSHOT_TARGET=list|detail|dialogs|calendar|tags|mobile|all` を指定する。
特定サイズだけ直接指定する場合は、`ARTICLESHELF_SCREENSHOT_VIEWPORT=desktop|macbook|laptop|tablet|mobile|mobile-md|mobile-sm` を使う。

## 6. 受け入れ条件

- すべての標準確認サイズで、主要操作が画面外に消えない。
- 検索、並び替え、フィルタ、追加ボタンの文言が見切れない。
- 記事カードのタイトル、概要、タグ、日付、ステータス、操作ボタンが重ならない。
- 詳細の閲覧 / 編集切り替えで、保存、削除、戻る、メタ情報が見切れない。
- スマホでボトムナビが一覧末尾、モーダル、確認操作を邪魔しない。
- カレンダー、タグ管理、フィルタ、追加、削除確認がタブレット / スマホで操作できる。
- `npm run capture:responsive` で標準確認サイズごとの診断スクリーンショットを分割出力できる。
